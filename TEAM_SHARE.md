## 节 1 — 起因(4 min)

### 开场

> "我最近做了个小工具叫 Agent Harness, 想跟大家分享一下。
>
> 先说为什么做。
> 我用 Claude 写代码有一段时间了, 反复遇到几个问题。
> 这些问题不是 Claude 笨, 我相信大家也都碰到过, 我先列一下,
> 看看是不是普遍现象。"

### 问题 ①:一句话需求, AI 起手 200 行

我跟 Claude 说"加一个密码强度评分功能"。
我心里想的是一个 30 行左右的纯函数, 输入密码、输出 0-100 分。

它给我写了 200 多行:6 个评分维度、3 个 strategy 类、一个 plugin 系统, 还配了 README。

我看了一眼觉得"嗯, 挺全的", 就接受了。
一周后我要改一个边界条件, 翻进去, 才发现这 200 行里我真用上的只有 30 行。
剩下的都是它"猜"我可能想要的功能。

> 反思一下: 不是 Claude 写多了, 是我给的需求太宽,
> 没说清楚就让它动手了。

### 问题 ②:每次新会话, 都要重新介绍项目

我们项目的背景信息其实不复杂:Gin + 四层架构、错误码统一从 errx 包出、测试用 table-driven。
但每次开新 Claude, 我都要再讲一遍。

讲完, 它才能开始干活。
一周下来, 这段话我可能讲过五六次。
重复输入这件事本身不累, 累的是**每次得花几分钟把脑子里的项目背景重新组织成一段话**。

> 反思一下: 这些信息明明可以写在文件里, AI 自己读就行。
> 但我之前就是没建立这个习惯, 也没工具帮我建立。

### 问题 ③:跟 AI 协作 2 小时,我自己什么都没干

下午 2 点开始, 跟 Claude 一起写一个功能。
我看着它每一次 tool call, 它写一段我看一段, 不对就纠正。
4 点抬头, 才发现这两小时我没做任何独立思考的事 —
全程都在监督它。

它好像一直在产出代码, 但我自己作为工程师的"判断力", 这两小时贡献几乎为零。

> 反思一下: 我把"实时纠错"当成了"协作", 其实是当了它的监工。
> AI 帮我节省的时间, 被我监工它的时间抵消了大部分。

### 问题 ④:AI 自信地告诉我它用了一个不存在的 API

它说: "我已经实现了 X 函数, 用了 Y 库的 Z API。"
听起来很专业。我看了眼代码, 也确实调用了 `Y.Z(...)`。
跑起来才发现 Y 库根本没有 Z。AI 编了一个。

更让我后怕的是: **我审过的代码, 我审到了什么程度?**
之前所有"看起来对就 commit"的地方, 是不是也有类似的问题, 只是还没暴露?

> 反思一下: AI 用同样自信的语气讲对的和错的, 我又没有靠谱的核验机制。

### 收尾

> "这 4 个问题里, 我自己反复中招最多的是第 1 和第 3 个。
>
> harness 这个工具, 是我针对其中几个问题做的一些工程化处理。
> 不是说能根治, 是说能把发生频率明显降下来, 也帮我自己改了一些用 AI 的习惯。
>
> 接下来 25 分钟我讲三件事:这工具是啥、怎么用、对应这 4 个问题怎么治。"

### 讲稿要点

- 整节用第一人称("我")多, 第二人称("你")少 — 是分享我自己的反思, 不是指控听众
- 节奏平稳, 每个问题 50 秒, 不用排比、不用强调句
- 收尾说清楚 **不夸大效果**:"不是根治, 是降低频率"

## 节 2 — harness 是什么(2 min)

### Slide

```
Agent Harness

  本质:一份 Python 脚本 + 一组 markdown 模板
  装的位置:你项目根目录的 .harness/ 和 .claude/
  装一次:5 秒

  装上之后做三件事:
    1. 给 Claude 一套固定工序 (设计 → 写测试 → 写实现 → 验证)
    2. 让 Claude 每次开会话/派子代理, 自动从你的 spec 文件读上下文
    3. 提供一句话触发: "按 design.md 开发"

  今天不讲它内部怎么实现的 (hooks / Teams API / state machine 这些)。
  只讲怎么用。
```

### 讲稿要点

- 它不是 IDE, 不是 Cursor 插件, 不是 Claude 的 fork
- 一句话定位:**它是 Claude 在你项目里的项目经理 + 工序卡 + 团队规范守护者**
- 我故意做得很小 — 一个 Python 脚本就完事, 没引入任何新依赖

### 一图(slide)

```
┌────────────────────────────────────────────────────────┐
│ 你: "按 design.md 开发"                                │
└────────────────┬───────────────────────────────────────┘
                 │ 自动触发
                 ▼
┌────────────────────────────────────────────────────────┐
│ 主会话(你 + Claude)                                    │
│                                                         │
│   ┌──────────┐   ┌───────────┐   ┌──────────┐          │
│   │ 架构师   │──▶│ 测试员    │──▶│ 开发者   │          │
│   │          │   │ 写失败    │   │ 让测试   │          │
│   │ 切模块   │   │ 测试      │   │ 通过     │          │
│   └──────────┘   └───────────┘   └──────────┘          │
│                       ▲                                 │
│                       └─── 测试员(再来跑一次 + 边界)   │
│                                                         │
│   每个 slice 完成 → 你 review → 主会话 commit           │
└────────────────────────────────────────────────────────┘
```

## 节 4 — 怎么用(12 min)

### 4a. 一行命令装好(1 min)

```bash
$ python3 init-harness.py --target .

✓ .harness/ 目录建好
✓ Claude hooks 注册
✓ 3 个角色 agent 准备好
✓ 触发 skill (harness-implement) 就位
✓ 追问 skill (grill-me) 就位
✓ RTK 装好 (Bash 操作 token 省 60-90%)
✓ Caveman 装好 (输出更紧凑)
✓ .gitignore 增量追加

完成。你现在可以说: "按 design.md 开发"
```

#### 讲稿要点
- 一条命令, 装上一整套
- RTK 和 Caveman 这两个 token 优化工具顺手装上, 省得大家自己装
- 不会覆盖你已有的 `CLAUDE.md` / `.gitignore` / `settings.json`,只增量追加
- 加 `--no-rtk` / `--no-caveman` / `--check-deps` 可以控制行为

### 4b. 一句话触发(1 min)

```
你不需要记一长串 prompt。

项目根有 design.md, 跟 Claude 说这些话之一就够了:

  • "按 design.md 开发"
  • "implement design.md"
  • "用 harness 实现这份设计"

整套流程就启动了。
```

#### 讲稿要点
- 我之前自己用的时候, 启动 prompt 写到 500 字, 实在记不住
- 后来包成 skill, 就一句话触发
- 我没有 magic, 只是把那 500 字塞到了 skill 的描述里

### 4c. 3 个角色, 各干各的事(3 min)

```
                ┌─────────────────────────────────────┐
                │ 架构师                               │
   prd.md ─────▶│   读: PRD + 团队规范                │
                │   出: info.md                       │
                │       (模块切分 / 接口契约 /        │
                │        slice 顺序 / 风险点)         │
                └─────────────────┬────────────────────┘
                                  │
                ┌─────────────────▼────────────────────┐
                │ 测试员(写失败测试)                  │
   info.md ────▶│   读: PRD + info.md + 测试规范       │
                │   出: 一组失败的测试                 │
                └─────────────────┬────────────────────┘
                                  │
                ┌─────────────────▼────────────────────┐
                │ 开发者                               │
                │   读: PRD + info.md + 失败测试       │
                │   出: 让测试通过的最少代码           │
                └─────────────────┬────────────────────┘
                                  │
                ┌─────────────────▼────────────────────┐
                │ 测试员(跑测试 + 找边界)            │
                │   读: 当前代码 + git diff            │
                │   出: 测试结果 + 边界用例 + 验收报告 │
                └──────────────────────────────────────┘
```

#### 讲稿要点

- **架构师** 不是写需求的, 是把你的 design.md 翻译成 AI 能照着实现的技术分解。它产出 info.md, 包括模块划分、接口契约、slice 顺序、风险点
- **测试员先于开发者出场**:这是流程里我自己改了好几版才稳下来的。逻辑是 — 没有失败测试, 开发者不许动;没有跑过测试, 代码不许 commit
- **开发者** 只为了让现有测试通过写代码, 不许多写
- 这 3 个角色都是 Claude 的 sub-agent, 你看不到它们写代码的细节, 只看到主会话汇总:
  > "slice 1 完成, 12 个测试通过, 代码 80 行。我可以 commit 吗?"

### 4d. 你只做 3 次决策(2 min)

```
整个流程, 你只在 3 个地方出场:

  ① "我把 design.md 翻成了 prd.md, 你看看准不准?"

  ② "架构师切了 5 个 slice, 顺序 X→Y→Z, 你 OK 吗?"

  ③ "slice N 完成, 12 个测试通过, 代码 80 行。我 commit 吗?"

其他时间, AI 自己跑。

──────────────────────────────────────────

  我的建议:
  PRD 确认完, 先放下电脑去做点别的事 (比如喝杯水、看会儿邮件)。
  30 分钟后回来看架构师的输出。

  我自己刚开始用的时候, 也忍不住一直盯着,
  花了几次才慢慢改掉这个习惯。
```

#### 讲稿要点

- 这 3 个 checkpoint 是写在 skill 里的, AI 会主动停下来等你, 不会跳过
- 实话实说:**改"忍不住盯着 AI"这个习惯, 比装这个工具更难**
- 如果你只是装了 harness 但还是全程盯着, 它的价值发挥不出来

### 4e. PRD 是输入, 不是 AI 帮你想(3 min) ★ 最关键

```
回想一下问题 ① — 一句话需求, AI 起手 200 行。

那不是 AI 写多了。
是我把"想清楚需求"这件事甩给了它。

它没有上帝视角, 没有团队上下文, 只能在它训练数据里
最常见的实现里挑一种给我。
我给的需求越宽, 它挑得离我真实意图越远。

更麻烦的是 — 它写出来 200 行, 我大概率会接受。
因为推翻这 200 行, 比我重写 30 行还累。

──────────────────────────────────────────

harness 把这件事翻过来了:

  ✗ "AI, 帮我设计一下 X 功能, 然后写"
  ✓ "我有一份 design.md, AI 帮我实现它"

PRD 是用户输入。AI 不允许重写它。
它只做翻译、切片、实现这些"二次加工"。

──────────────────────────────────────────

那如果我真的写不细呢?
我做了一个配套的 skill 叫 grill-me。

你写 1 行: "加一个登录功能"
grill-me 会主动追问:
  - 登录失败返回什么状态码?
  - 要不要 rate limit? 多少次后封?
  - 错误信息要不要本地化?
  - session 多久过期?

它问到你说"够了"为止。
问完, 你的需求从 1 行扩成 30 行可执行的 spec。

  这一步最难的不是工具, 是让自己习惯
  "先想清楚, 再让 AI 动手"。
  我自己也是磨了好几个项目才慢慢做到。
```

#### 讲稿要点

- 这是整套 harness 里我自己最深的一个认知转变
- **写 spec 是用户的活, 不是 AI 的活** — 这个观念跟很多人对 AI 的初始预期是反的, 需要点时间适应
- **现场示意**:打开 grill-me 在另一个项目的真实问答记录(预先准备的截图)

### 4f. 让上下文从文件来,不从你嘴里说(2 min)

```
回想一下问题 ② — 每次新会话都要重新介绍项目。

那是因为项目信息只存在两个地方:
  ① 你脑子里
  ② 你跟 AI 说过的某段对话里 (它已经忘了)

──────────────────────────────────────────

harness 让你把这些东西沉淀到文件里:

.harness/spec/index.md       # 团队代码规范
.harness/spec/testing.md     # 测试约定
.harness/spec/errors.md      # 错误处理风格

写一次, 长期复用。

然后每个任务里, 你 curate 一份"上下文清单":

context.architect.jsonl
  → 派架构师时, 自动带上这些文件

context.developer.jsonl
  → 派开发者时, 自动带上这些文件

context.tester.jsonl
  → 派测试员时, 自动带上这些文件

──────────────────────────────────────────

  写到文件里 AI 每次都能读, 比口头交代靠谱很多。
  写 spec 这件事是一次性投入, 长期省力。
```

#### 讲稿要点

- 新人最容易翻车的地方:**不 curate 这个清单**
- harness 故意设了一个挡板:**不 curate 任务不让开工**
- 这个摩擦是有意设计的 — 我自己用的时候发现, 不强制就会偷懒, 偷懒了 AI 就拿不到正确上下文
- curate 不是把所有文件都列上, 而是想清楚"这个角色这个 slice 真正需要看什么"

## 节 5 — 这几个问题, harness 是怎么对付的(4 min)

> 把开头那 4 个问题, 一个一个对应过来, 说清楚怎么治、有什么局限。

### 问题 ①:AI 起手就 200 行

**怎么对付**
- design.md 是你写的, AI 不许重写
- grill-me 把模糊需求问明白
- 架构师切 slice, 你能指着说"这 5 个我只要 1 和 2"

**还存在的局限**
- 如果你 design.md 自己就写得很乱, harness 帮不了你太多
- grill-me 问得对不对, 也依赖你回答得有多认真

**实操经验**
- 任何一个 slice 实现超过 100 行, 我会让架构师重切

---

### 问题 ②:每次新对话要重新介绍项目

**怎么对付**
- 任务存在 .harness/tasks/, 关掉 Claude 也不丢
- 每次开新会话, hook 自动告诉 AI 当前任务和状态
- 项目规范写在 spec 文件里, 上下文清单按需引用

**还存在的局限**
- 第一次写 spec 文件还是要花点时间, 这是一次性投入
- 团队里如果有人不维护 spec, 这套机制会逐渐失效

**实操经验**
- 我建议先在自己一个小项目上试, 把 spec 沉淀好之后再推给团队

---

### 问题 ③:跟 AI 协作 2 小时,自己什么都没干

**怎么对付**
- 把"实时纠错"压缩成 3 次批量决策
- 其他时间它后台跑

**还存在的局限**
- 这是工具能做的部分。剩下的要靠你自己改"想盯着它"的习惯
- 工具不能强迫你离开屏幕

**实操经验**
- PRD 确认完之后, 我会主动找点别的事做, 30 分钟回来看一次
- 如果你装上 harness 但还是全程盯着, 那效果跟没装差不多

---

### 问题 ④:AI 编造 API、库名、函数

**怎么对付**
- 测试员先写失败测试 → 开发者只为让测试通过写代码
- commit 之前必须看到测试 GREEN 的真实输出, 不是"AI 说通过了"

**还存在的局限**
- 测试本身写得对不对, 还是依赖 architect 和 tester 的协作质量
- 编造问题在生产代码里能拦下来, 但聊天闲聊里 AI 还是可能瞎说

**实操经验**
- 凡是 AI 说"我已经实现了 X", 我都让它把测试输出贴出来再说
- 不贴 = 不信

## 节 6 — 怎么接入(2 min)

```
☐ cd 到你项目根目录

☐ python3 init-harness.py --target .

☐ 跟同事约定:
   把 design 文档放到项目根 (design.md)

☐ .harness/spec/ 里写团队规范
   一次写好, 反复复用

☐ 试跑一个小 feature
   不要拿核心模块开手 — 选一个有限风险的子命令最好

☐ 在 3 个 checkpoint 认真 review

☐ 第一次跑完, 翻一下 .harness/tasks/<task>/ 里每个文件
   理解每份文件做什么
```

### 讲稿要点

- 第 4 步是关键 — spec 写得越好, AI 写代码越准。这是一次性投入
- 第 5 步建议:**第一次试跑选个低风险的 feature**, 不要拿核心模块开手
- 第 7 步建议作为"分享会作业":回去之后跑完一遍, 翻一下产物, 比听我讲 30 分钟更有用

## 节 7 — 看看 demo 跑成啥样了(2 min)

### 现场动作

切回前面那个 Claude, 看:
- `.harness/tasks/` 里的产物
- 主会话最后一条消息(应该是"全部 slice 完成, 等你 review")
- `git log` 看 commit 数和粒度
- 跑测试看是否 GREEN

### 讲稿要点

- 平实讲:"这是它跑的产物, 我们一起看一下"
- 跑完了:"它走完了 4 个 slice, 通过了 N 个测试, 给了 4 个 commit"
- 没跑完(在 checkpoint 等):"它在等我 review, 这是流程设计的一部分, 不是卡死"
- 出错了(不太常见但有可能):**坦诚说出来** — "这一步它没拿到正确的 X, 这种情况怎么排查 …"

### 一句客观评价

> "你看到的这个产物, 跟我自己手写比, 优点是流程齐、测试齐、风险点列出来了;
> 缺点是它的代码风格是 AI 风格, 偶尔会有过度抽象的地方需要我再 review 一遍。
> 所以我前面说 — 这工具不是替代你写代码, 是帮你把工序固化下来。"

## 节 8 — Q&A + 不适合的场景(2 min 缓冲)

### 不适合用 harness 的场景(主动说清楚)

```
☓ 探索性脚本 (你自己也不知道要写啥, 边写边想)
☓ 一次性临时代码 (用完就扔)
☓ 复杂遗留系统的核心改造 (上下文太多, 工具兜不住)
☓ 不想写 spec 的项目 (强行用反而更慢)

✓ 中小型、有清晰 spec 的功能开发
✓ 团队规范固定、复用性强的项目
✓ 对代码质量要求 > 速度的场景
```

### 预设可能问题

| Q | 简短答 |
|---|------|
| 跟 Cursor / Cline 比? | 它是 Claude Code 上的脚手架, 不替换 IDE。可以同时用 |
| 跟其他 skill 冲突? | 不冲突。harness 管"工序", 其他 skill 管"技法", 互补 |
| 多人协作怎么办? | `.harness/` 进 git, 任务状态共享, 但目前不支持并发 task |
| token 消耗大不大? | RTK 会装上, Bash 操作能省 60-90%。整体比"长 prompt 反复粘"省很多 |
| Claude 版本要求? | Claude 4.x / 4.7 都能用 |
| 现在还有什么明显短板? | (照实说) Sub-agent 之间的状态同步还要靠 commit, 不能并发跑同一个 slice |
